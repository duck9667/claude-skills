# Management Mental Models

의사결정·매니지먼트·조직 운영 대화에서 사고 파트너 역할을 할 때 쓰는 렌즈 모음.
`references/`에 원문 노트 5개가 있고, 이 문서는 **어떤 상황에 어떤 렌즈·문서를 꺼낼지**를 안내한다.

## 적용 원칙

- **판단성 주제**(트레이드오프, 조직/팀 문제, 우선순위, 지표 설계 등)면 맞는 렌즈를 자연스럽게 꺼내 검토를 돕는다. 프레임워크 이름을 나열하는 게 아니라, 렌즈로 본 결과를 대화에 녹인다.
- **명시 요청**("멘탈모델로 봐줘", "프리모템 해줘", "시스템 사고로 분석해줘")에는 해당 도구를 정식으로 적용한다 — 이때는 관련 reference를 읽고 깊이 있게.
- 사용자의 판단에 **과신·구조 문제·잘못된 지표** 같은 패턴이 보이면 지적한다.
- 단순 작업 요청에는 강제하지 않는다.
- 렌즈끼리 결론이 갈리면 그것 자체가 정보다("문제가 다층적이다") — 억지로 하나로 수렴시키지 않는다.

## 핵심 어휘 (요약 — 자세한 건 references)

| 렌즈 | 한 줄 | 출처 |
|---|---|---|
| **스톡 / 플로우** | 누적(신뢰·평판·기술부채)과 속도(이번 달 매출)를 분리해서 봐라 | 메도즈 |
| **피드백 루프** | 폭주 → 강화 루프, 끈질긴 제자리 → 균형 루프 | 메도즈 |
| **지연 → 진동** | 시간차를 무시한 과잉 반응이 출렁임을 만든다 | 메도즈 |
| **지렛점** | 숫자(얕음) < 정보 흐름·규칙 < 목표·패러다임(깊음). 단 깊을수록 안 움직임 | 메도즈 |
| **레버리지** | 한 시간이 만드는 2~3차 효과. 많은 사람에게 + 오래 + 결정적 시점에 | 그로브 |
| **one-way / two-way door** | 되돌리기의 *실질 비용*으로 분류. 한쪽 문은 느리게·깊게, 양쪽 문은 빠르게 | 종합 |
| **자유 토론 → 명확한 결정 → 전적인 지지** | 동의는 토론 단계의 일, 지지는 실행 단계의 일 | 그로브 |
| **TRM (과업별 성숙도)** | 사람이 아니라 사람×과업 단위로 지시→코칭→위임 | 그로브 |
| **고통 + 성찰 = 발전** | 단, *감당 가능한* 고통일 때만. 성찰의 산출물은 기분이 아니라 규칙 | 달리오 |
| **믿음도 가중** | 직급이 아니라 그 주제의 실적+논리로 의견에 무게. 단, 권력 오염 주의 | 달리오 |
| **pre-mortem** | "6개월 뒤 실패했다고 가정 — 뭐가 잘못됐나"를 결정 *전에* 묻기 | 종합 |
| **확률로 결정하기** | "잘 될 것 같다" 대신 "성공 확률 70%, 성공이면 Y, 실패면 Z가 관찰될 것" | 종합 |
| **심리적 안전감** | 솔직함·실수 보고·반대 의견의 *전제 조건*. 명령이 아니라 반복된 경험으로 쌓임 | 2026 |
| **옵션성 / 안티프래질** | 확률을 모르는 불확실성 영역에선 비대칭 셈법(작게 잃고 크게 얻을 옵션) | 2026 |
| **권력 문해력** | 시스템 분석과 별도로 "누가 이 결정으로 권력을 얻나/잃나"를 한 번 더 본다 | 2026 |
| **암묵지** | 체크리스트로 못 옮기는 영역이 있다. 도구를 말 그대로 복사하면 깨진다 | 2026 |

## 주의

- 문서들은 원서의 재구성 해설 노트다. 세부 인용이 필요하면 원서 확인을 권하라.
- 2026 노트(5번)는 1~3번 문서의 일부 주장(극단적 투명성, 공장 메타포 등)을 **의도적으로 반박**한다. 두 관점이 충돌하면 충돌 자체를 사용자에게 보여주는 것이 가치다.
- 가장 큰 함정은 하나의 렌즈를 모든 상황에 적용하는 것. 렌즈는 답이 아니라 질문을 정확하게 잡는 도구다.
